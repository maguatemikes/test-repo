# Migrations

Raw MySQL migrations for the `omnc` database, applied in numbered order.

Two parallel migration paths exist in this repo:

1. **Drizzle migrations** (in `./drizzle/`, generated by `pnpm db:generate`) — for ongoing schema evolution of the new CRM tables once the project is live.
2. **Manual SQL migrations** (in this folder) — for the initial setup, the NetX retrofit, and the baseline seed. These exist because they need explicit operator review and coordination with the NetX desktop team.

After the initial three files in this folder are applied, day-to-day schema changes happen through Drizzle (`pnpm db:generate` → review → `pnpm db:migrate`).

## Application order

```
0001_create_crm_tables.sql        # Required first. New tables only. Safe to run anytime.
0002_netx_org_id_retrofit.sql     # Adds organization_id to existing NetX tables. COORDINATE.
0003_seed_baseline.sql            # Seeds plans + the internal organization row.
```

### How to apply

From your local machine (with mysql CLI installed and DB credentials):

```bash
mysql -h 216.219.83.61 -P 3206 -u <user> -p omnc < migrations/0001_create_crm_tables.sql
mysql -h 216.219.83.61 -P 3206 -u <user> -p omnc < migrations/0002_netx_org_id_retrofit.sql
mysql -h 216.219.83.61 -P 3206 -u <user> -p omnc < migrations/0003_seed_baseline.sql
```

Or from the database server itself (SSH in first):

```bash
ssh <user>@216.219.83.61
mysql -u <user> -p omnc < /path/to/0001_create_crm_tables.sql
# ...
```

## What each migration does

### `0001_create_crm_tables.sql`

Creates 27 new tables under the following groups:

- **Tenancy:** `plans`, `organizations`, `users`, `organization_members`, `api_keys`, `usage_meters`
- **Customers:** `customers`, `customer_tags`, `customer_tag_links`, `customer_addresses`, `customer_aliases`
- **Audience:** `lists`, `list_members`, `segments`, `segment_members`
- **Email infra:** `sending_domains`, `senders`, `templates`, `suppressions`
- **Sending:** `campaigns`, `emails`, `email_events`
- **Automation:** `automations`, `automation_steps`, `automation_runs`
- **Forms:** `forms`, `form_submissions`
- **Integrations:** `integrations`, `webhooks`, `audit_logs`

All CREATE TABLE statements use `IF NOT EXISTS`, so this file is safe to re-run.

**No impact on NetX** — these are brand-new tables.

### `0002_netx_org_id_retrofit.sql`

Adds an `organization_id BIGINT NOT NULL DEFAULT 1` column to every NetX customer-touching table:

- `netx_orders`
- `netx_customer_meta`
- `netx_customer_meta_history`
- `netx_customer_alias`
- `netx_marketplaces`
- `netx_marketplace_content`
- `netx_video_calls`
- `netx_video_operators`

Each ALTER is guarded by an `INFORMATION_SCHEMA.COLUMNS` check (same pattern used in `netx_updater_v1_schema.sql`), so the file is safe to re-run.

**Why `DEFAULT 1`:** NetX desktop code currently doesn't know about `organization_id`. With a default of 1, any new INSERT from the existing NetX code continues to work, with rows automatically tagged to the internal store. Once NetX is updated to explicitly pass `organization_id`, the default is removed in a Phase 2 migration (template at the bottom of the file).

**Coordination required:**

- Tell anyone using NetX desktop that orders, customer meta, and marketplace records get a new `organization_id` column. Most code won't notice because of the default.
- Tell the .NET API codebase (when it starts) that every read must filter by `organization_id`, and every write must specify it.

### `0003_seed_baseline.sql`

Inserts baseline data:

- **4 plan rows** (Free, Starter, Growth, Enterprise) with realistic limits and feature flags
- **The internal organization** at `id = 1`, slug `internal`, linked to the Enterprise plan

The plans use `ON DUPLICATE KEY UPDATE` so you can tweak `slug`/`name`/limits later by re-running this file. The internal organization insert is guarded by an existence check.

**Adjust before running:** change `'Internal Store'` to your preferred organization name in the file before applying — that's the name that will appear in the org switcher in the web app.

## Verifying success

After all three migrations apply, check:

```sql
-- Should return 27 rows
SELECT COUNT(*) AS new_tables
  FROM INFORMATION_SCHEMA.TABLES
 WHERE TABLE_SCHEMA = 'omnc'
   AND TABLE_NAME IN (
     'plans','organizations','users','organization_members','api_keys','usage_meters',
     'customers','customer_tags','customer_tag_links','customer_addresses','customer_aliases',
     'lists','list_members','segments','segment_members',
     'sending_domains','senders','templates','suppressions',
     'campaigns','emails','email_events',
     'automations','automation_steps','automation_runs',
     'forms','form_submissions',
     'integrations','webhooks','audit_logs'
   );

-- Should return the internal organization
SELECT id, slug, name, status FROM organizations;

-- Should return 4 plan rows
SELECT slug, name, price_monthly_usd FROM plans ORDER BY price_monthly_usd;

-- Confirm NetX orders all have org_id = 1
SELECT organization_id, COUNT(*) AS rows
  FROM netx_orders
 GROUP BY organization_id;
```

## Drizzle going forward

For every schema change after the initial setup:

1. Edit `src/lib/db/schema.ts`
2. Run `pnpm db:generate` — Drizzle inspects the diff and writes a new SQL file under `drizzle/` (e.g., `drizzle/0001_yellow_starfox.sql`)
3. **Review the generated SQL** — Drizzle sometimes generates destructive operations you didn't intend
4. Apply with `pnpm db:migrate` (or manually with `mysql < drizzle/0001_*.sql` if you want explicit control)
5. Commit both the updated `schema.ts` AND the generated migration file

## Production safety checklist

Before applying any of these to the production database:

- [ ] Take a `mysqldump` backup of `omnc` first
- [ ] Confirm the connection user has CREATE TABLE, ALTER TABLE, and INDEX privileges
- [ ] Apply during a low-traffic window if NetX is actively writing
- [ ] Run on a staging copy first if you can spin one up
- [ ] Verify with the post-migration queries above before declaring success
- [ ] Keep the backup until the next backup cycle completes successfully
